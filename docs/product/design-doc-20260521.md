## Problem Statement

创始人想做一款面向"想养猫/刚养猫的纯新手"的产品。原 PRD 定位为"一站式答疑与档案管理工具",核心是带猫档案上下文的多轮 AI 问诊 + AI 归纳的场景化手册。

经 office hours 诊断,真实问题被重新定义:创始人(及假设中的目标用户)的核心痛点不是"找不到答案",而是 —— **面对猫的健康/行为/生活问题时,搜到的答案互相矛盾、无法分辨真假,由此产生的不确定性与焦虑**。创始人原话:"答案可能都不一样,无法分辨是真的还是假的""主要担心小猫生病。"

因此产品的真正任务,从"答疑工具"被重构为"**帮一个怕做错、怕害了猫的新手做出可信判断、并因此安心**"的决策伴侣。

## Demand Evidence

诚实记录:**当前需求证据等级为 n=1(仅创始人本人)。**

- 创始人是自己的用户:日常养猫中反复遇到该痛点,举出具体例子 —— 什么花有毒、什么水果猫能/不能吃、猫眼睛流泪、打喷嚏等,需逐个去百度/小红书搜索并交叉比对。
- Q2 现状代价:创始人选择"主要是时间" —— "翻多个来源、交叉比对很费时间,但最后总能凑出结论"。即:高频低痛(多数问题靠搜索能解决)+ 低频高痛(如"百合致命"这类小概率灾难性问题)。
- Q3 关键缺口:创始人有养猫的朋友,但**从未问过他们的痛点**。所有判断建立在一个数据点上。
- 无人付费、无人催问、无候补名单、无任何外部需求行为证据。

**结论:这是一个有真实个人痛点的种子,但尚未被验证为他人也强烈需要的需求。这是当前最大的风险,必须在投入认真开发前/中解决。**

## Status Quo

目标用户当前的解决方案:**免费**。百度、小红书、抖音搜索,自行交叉比对;有 VPN 能力的少数人可能用 ChatGPT,但创始人指出国内大众用户基本不挂梯子、也不习惯用豆包/DeepSeek。

代价:主要是时间成本(高频);以及"比对完仍不确定"的焦虑(创始人未将其计为主要代价 —— 说明该痛点目前烈度中等偏下,尚未造成可量化的损失)。

关键事实:现状是免费的、且"基本够用"。要让用户切换到新产品并填写猫档案,产品必须在某个点上明显优于免费方案。

## Target User & Narrowest Wedge

**目标人物(待真实访谈替换 —— 当前为假设):** 刚把猫接回家头几周、对猫一窍不通、对"做错会害了猫"高度焦虑的纯新手。注意存在画像错配:创始人本人可能已不属于此画像(创始人是有经验、知道该问什么的猫主),而 PRD 核心交互"用户主动发问"恰是为"知道问什么的人"设计的。

**最窄楔子:** "症状/情况分诊器" —— 当用户对猫的某个情况(健康、行为、绝育时机等)感到不确定时,产品通过 3-6 个关键追问 + 猫档案,输出**风险分级 + 下一步动作 + 判断理由**(红=立刻就医 / 黄=24h 观察+护理 / 绿=低风险处理),而非一段"仅供参考"的百科文本。

**组织原则的转变(本设计的核心):** 从 PRD 的"8 个并列场景的话题目录"(= 维基,与百度/小红书拼词条且必输)转为"用户在某个时刻的不确定 → 帮 ta 决策"(= 决策伴侣 = 有留存的安心产品)。生活类话题(行为异常、绝育时机、换粮方法)仍可纳入,但组织方式是"帮你判断",不是"给你词条"。

## Constraints

- 单人开发,创始人有技术能力。
- MVP 周期:方向 A 的构建估计 CC 辅助 ~1 周 / 单人无辅助 ~2-3 周;叠加用户验证轨道后,诚实的总周期约 3-4 周(见 Recommended Approach),不是原 PRD 设想的"4 周做完一切"。
- 目标用户为国内、移动端为主、搜索型(非 chatbot 型)用户。
- 明确排除(沿用 PRD 并强化):多猫管理、社区/UGC、电商/导购、真人在线问诊。
- **新增约束(本次):** 不做"猫粮品牌推荐"(信任最难,且唯一变现路径是导购,与"不做电商"自相矛盾);不把低安心价值的"填充型"功能(如玩具偏好)当作核心。
- 安全:医疗相关回答固定附"AI 不能替代兽医";红线症状触发即弹送医模板。

## Premises

经 office hours 确认(P2 经创始人反驳后修订;P1/P3/P4 创始人认同;Codex 独立判定四条全部成立):

- **P1(认同):** 这是情绪/安心类产品,不是答疑工具类产品。工具类与 ChatGPT 硬拼,情绪类才有留存和口碑。
- **P2(修订):** 创始人 PRD 点名的两个差异点("猫档案上下文""AI 整理的知识库")不构成壁垒 —— 国内已有同样功能的产品(千珍万宠等),且小红书/抖音以社区真人信任在情绪轴上结构性占优。创始人的反论(用户是搜索型、活在小红书)是有效的**分发洞察**,属渠道层面,不是功能壁垒。
- **P3(认同 —— 本设计硬约束):** 产品要解决"信任",但 PRD 原方案("LLM 整理审核的知识库" + "内容由 AI 整理,仅供参考"的免责声明)解决不了信任。一个标注"仅供参考"的答案,无法让一个本就不信网上答案的人安心。**缓解机制:** "分诊"框架将信任门槛从"每个答案都要正确"降为"不要漏掉急症",一个保守设计(拿不准即偏向送医)的分诊器可达到后者;静态参考卡(有限、可被兽医逐字核对)进一步降低信任风险。**但完全解决信任仍需兽医署名(见方向 B)。**
- **P4(认同):** 创始人选择创业模式,但路径(不做电商、远期订阅制)避开了该品类唯一被验证的变现方式。Codex 指出 P4 最被低估,建议用"支付实验"而非访谈来验证变现。

## Cross-Model Perspective

Phase 3.5 由 Codex(gpt-5.5)进行独立冷读(未参与本次对话,仅获结构化简报)。要点:

1. **最强版本:** 这不是"猫知识问答 App",而是"新手猫主的安心操作系统" —— 在用户慌乱时,把模糊症状、猫档案、风险等级、下一步动作压缩成一个可信决策。
2. **最关键原话:** "主要担心小猫生病。"真实需求是降低不确定性下的焦虑,不是查资料。
3. **Premise 判断:** 四条基本都对,P4 最被低估;建议用支付实验(同流量测试多个付费选项,如 9.9 元一次分诊报告 / 29 元 7 天陪跑 / 99 元订阅)验证变现,而非再做访谈。
4. **48 小时原型建议:** Next.js + Supabase + Vercel + Claude Sonnet;不做 pgvector、不做复杂知识库、不做原生 App,先做移动端 Web;只做猫档案 + 症状分诊流 + 红黄绿安心报告;砍掉多轮闲聊、完整手册、社区、电商、提醒、真人兽医。

跨模型一致性:Claude 与 Codex 在核心诊断(情绪/安心产品、"担心生病"是内核、四条 premise 成立)上完全一致。分歧仅在节奏:Codex 倾向直接做原型 + 支付实验;Claude 强调原型能验证"是否付费",但验证不了"用户到底怕什么、用什么词",故主张原型与用户访谈并行。

## Approaches Considered

三个方向共享:分诊器(骨架)+ 猫档案(喂数据)+ 一小叠静态参考卡(原"手册"的提纯版:有毒植物卡、安全食物卡、疫苗驱虫时间线等,约 10-15 张,静态、可被兽医逐字核对)。

### Approach A: 纯分诊器 MVP(最小可行)—— 【创始人选定】
- Summary: 只做分诊流 + 猫档案 + 一小叠静态参考卡(MVP 先做 3-5 张最高风险卡:有毒植物、有毒食物、急症症状;其余 ~10 张延后)。移动端 Web,不做原生 App。**分诊引擎不是让 LLM 自由分类症状** —— 红线/急症走人工编写的决策树/规则(沿用 PRD 红线清单),LLM 只负责接收用户的自然语言描述、追问、和解释判断理由。这样既能在短周期内可靠落地,又避免在最关键的急症上出现幻觉。
- Effort: M(构建:CC 辅助 ~1 周 / 单人无辅助 ~2-3 周;另需前置 + 并行的用户验证轨道,见 Recommended Approach)
- Risk: Low-Med
- Pros: 最锐的楔子;绕开"答疑 vs ChatGPT"硬拼;移动端 Web 安装摩擦低;PRD 的红线症状清单从配角升为主角;以静态卡形式保留了创始人想要的"手册"。
- Cons: 在验证需求之前先建产品(n=1 未解);无分发方案;无兽医,信任只解决一半;**分诊校准(保守度)是核心难题,见 Open Questions 2**;单一分诊工具易被现有玩家复制。
- Reuses: 猫档案 schema、红线症状清单、8 场景(转为参考卡选题)、Next.js 技术栈。

### Approach B: 兽医署名的分诊器(理想架构)
- Summary: A + 找一位执业兽医付费审核分诊逻辑与参考卡,产品展示其姓名/资质,承诺从"AI 整理仅供参考"翻转为"经执业兽医审核"。
- Effort: M(人工 ~4-6 周,找兽医为长杆)
- Risk: Med
- Pros: 真正解决 P3 信任矛盾;可防御的信任位;"兽医署名"是 UGC 和纯 AI 产品不易复制的卖点。
- Cons: 慢;需建立尚不存在的兽医关系;持续审核成本;监管灰色需谨慎;仍是先建后验证。
- Reuses: A 的全部;PRD 知识库入库流程(改为兽医核对、有限)。

### Approach C: 先内容、后产品(创意/横向)
- Summary: 3-4 周不写 App 代码,改为访谈 8-10 个真实新手 + 把分诊内容发成小红书/抖音帖,验证需求、攒首批用户、用数据定范围,然后再建 A/B。
- Effort: 代码 S(零)/ 实操 M(人工 ~3-4 周)
- Risk: Low(可逆)
- Pros: 干掉 n=1;解决冷启动;近零成本验证需求;内容成为护城河;可逆。
- Cons: 不算"做产品",对建造者不爽;到"App 存在"更慢。
- Reuses: PRD 的 8 场景 + 知识库来源优先级转为内容日历。

## Recommended Approach

**Claude 的推荐是方向 C;创始人选定方向 A。两者都记录在案。**

创始人选择 A(先建后验证)是其作为建造者的合理判断,office hours 尊重该决定。但为对冲 A 的核心风险(n=1),A 的执行计划中**强制并入用户验证轨道**:

- **方向 A(执行版)= 纯分诊器 MVP + 前置 + 并行的用户验证轨道。**
- 用户验证不是建议,是约束,且部分前置于构建。诚实的总周期约 3-4 周:(a) **构建开工前(本周)** 完成 5-8 个真实新手访谈(见 The Assignment)—— 访谈结果是分诊追问设计的直接输入,必须先做;(b) 构建 ~1-3 周;(c) 原型可用后,坐在 3 位真实新手身后,看他们在无人帮助下使用分诊器,记录每一处困惑("看,别 demo")。
- 推荐后续路径:A(含验证轨道)→ 若验证为正,用方向 B 的兽医署名升级解决 P3 与分诊校准 → 变现用 Codex 建议的支付实验检验 P4(但支付实验需先有分发渠道,见 Open Questions 6、7)。

## Open Questions

1. 画像错配如何解决 —— 创始人本人是否真能代表"一窍不通的纯新手"?需用真实新手访谈校准。
2. 分诊校准 —— 这是整个产品最难的问题,远不止"医学正确性"。太保守(什么都让送医)= 退化成百度那种低信息量输出,还给本就焦虑的新手制造持续假警报,杀死留存;太松 = 漏掉急症、害猫。无兽医的 MVP 很可能无法独自把握这个平衡 —— 这强化了 P3"A 只解决一半信任"的判断,也是升级到方向 B(兽医署名)的真实理由。
3. 假阴性事故兜底 —— 若保守分诊仍把一个真实急症判为"绿",产品如何兜底?建议:记录并定期复核每一次"绿"判定;法律暴露的评估不能只靠"仅供参考"免责声明。
4. 监管边界 —— "分诊/风险分级"与"诊断"的法律界线;务必保持"建议是否就医",不做"诊断"。
5. 主力模型决策 —— 目标用户是国内、不挂梯子的移动端用户。Approach A 当前以 Claude Sonnet 为主、DeepSeek/通义为"兜底",但分诊是安全攸关逻辑,换模型需重新校准保守度,"兜底"并不便宜。建议:开工前就定主力模型;若发布受众确为国内用户,MVP 很可能应直接基于 DeepSeek/通义构建。
6. 变现假设 —— 创始人需在 MVP 后期就准备一个可被支付实验检验的变现假设,而非推到远期。注意:Codex 建议的支付实验需要流量,而分发(见下条)尚未解决 —— 支付实验应排在分发渠道建立之后,不是立即可执行的动作。
7. 分发 —— 移动端 Web 如何被搜索型用户发现?(与方向 C 的内容思路不互斥,可在 A 上线后启动。)
8. 数据合规 —— 猫档案、对话内容的存储与隐私协议,MVP 即需最简版本。

## Success Criteria

MVP(方向 A)成功的判据,不是"上线了",而是:

- 完成 5-8 个真实访谈,其中**至少 4 人必须是养猫 3 个月以内的真新手**(否则验证不了目标画像 —— 见 Open Questions 1);这些真新手中,至少 2-3 人能独立描述出与创始人相似的"不敢信、担心"痛点(否则 n=1 假设未通过,应暂停并重新评估)。
- 原型可用后,至少 3 位真实新手在无人帮助下完成一次分诊,并能说出"我会再用 / 我会推荐给同样新手的朋友"。
- 早期留存信号:有用户在不同问题上**第二、第三次**回来用分诊器。
- 失败止损线(默认线 —— 创始人可上调严格度,但不应放松):**访谈周结束时,若非创始人本人的真新手中少于 2 人独立描述出"担心/不敢信"痛点,立即暂停所有编码,回到需求验证。** 上线后:若 4 周内无任何用户自然二次回访,判定核心假设未通过。

## Distribution Plan

方向 A 的交付物是移动端 Web 应用(非原生 App,无应用商店分发)。

- 部署:Vercel + Supabase,沿用 PRD 选型(去掉 pgvector —— MVP 无需向量库,静态参考卡 + 分诊逻辑不需要 RAG)。
- 获客:本设计未解决分发。移动端 Web 对搜索型用户的可发现性是开放问题(见 Open Questions 7)。建议 A 上线后,借用方向 C 的思路 —— 把分诊卡/急救卡作为小红书/抖音内容投放,内容即获客入口。
- CI/CD:单仓 + Vercel 自动部署即可,MVP 阶段无需额外管线。

## Dependencies

- 5-8 位可访谈的真实新手猫主(创始人的朋友圈即可起步)。
- LLM API(Claude Sonnet 主;国内可用性需备 DeepSeek/通义作兜底)。
- 静态参考卡的内容来源(沿用 PRD 的权威源优先级:AAFP/ISFM 指南等)。
- (方向 B 才需要)一位愿意合作的执业兽医。

## The Assignment

**本周内,完成 5-8 个真实养猫者的访谈,其中至少 4 个必须是养猫 3 个月以内的真新手,每人 15-20 分钟。**

规则:不描述你的 App,不推销,只问、只记原话。三个问题:

1. 你上一次为猫的事担心、或专门查资料,是什么时候?具体发生了什么?
2. 你当时做了什么?去哪里查的?查的过程什么感觉?
3. 后来怎么解决的?现在回想,还有什么不放心的?

目标:找到第二个、第三个有"不敢信、担心"痛点的真实的人 —— 或者证明找不到。这一步的信息量,大于你接下来两周写的任何代码。它也直接为你的分诊器提供真实的症状用语和追问设计。

## What I noticed about how you think

- 你在被追问时,几次都选了**对自己不利但诚实的那个选项** —— "主要是时间""还没问过朋友的痛点"。多数人会美化。这种实事求是是创始人最稀缺、也最该保住的资产。
- 你反驳 P2 时说:"国内用户很多不会挂梯子……喜欢去百度小红书抖音搜索。"这是你整场对话里关于获客最敏锐的一句话 —— 你对用户的真实行为有直觉。但要分清:那是渠道洞察,不是功能壁垒。
- 你两次抵抗收窄("想保留手册""围绕人宠生活展开")。这是建造者的本能,但也是创业最常见的死法之一 —— 把范围当安全感。你最后接受了"宽话题、窄脊椎"的分法,这个判断是对的。
- 你说"主要担心小猫生病"时,大概没意识到自己刚说出了整个产品的心脏。你前面绕了"信任""时间"两圈,真相是"担心"。你的直觉一直对,只是被 PRD 的工具化语言盖住了。
