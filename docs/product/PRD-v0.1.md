# 猫咪新手一站式 App MVP 方案（讨论稿）

文档版本：v0.1（讨论用）
最后更新：2026-05-07

## 一、产品定位

### 一句话

面向"想养猫/刚养猫的纯新手"的一站式答疑与档案管理工具，核心是**带猫档案上下文的多轮 AI 问诊** + **AI 归纳的场景化养猫手册**。

### 起源痛点

新手养猫每次遇到问题都要去小红书、知乎、抖音搜索，且每次都要重复说明猫的情况（年龄、是否打针、绝育与否等），效率极低且答案良莠不齐。

### 与"直接问 ChatGPT/小红书"的差异

- **猫档案上下文**：系统始终知道用户问的是哪只猫、它的年龄/疫苗/到家时间等，回答精度高一档。
- **猫领域审核过的知识库**：通用大模型对"3 月龄能不能洗澡""疫苗后多久能驱虫"这类细节常给安全但模糊的答案；UGC 又良莠不齐。本产品的知识库经过整理与场景化分类。
- **多轮问诊式交互**：不是一问一答，而是像兽医一样追问关键信息后再回答，符合真实诊断逻辑。

## 二、目标用户

- 主画像：刚养猫或准备养猫、对猫一窍不通的新手。
- 使用阶段：以"已经把猫接回家"为主，"调研要不要养"为辅。
- 多猫家庭：MVP 暂不支持，留接口。

## 三、典型使用场景

- 焦虑实时答疑：半夜猫吐了，掏出 App 问"3 个月小猫吐黄水怎么办"。
- 提前学习：周末想系统了解疫苗周期、饮食过渡。
- 日常打卡：记录体重、疫苗、驱虫时间。

## 四、MVP 功能范围

### 必做（核心三件套）

#### 1. 猫档案模块

字段：
- 基础：昵称、品种、生日（年龄自动算）、性别、绝育状态、到家日期。
- 健康：疫苗记录（多条，含日期、品牌、剂次）、驱虫记录（多条，含体内/体外、药品）、体重曲线（多条，日期 + 体重）、过敏或慢性病备注。
- 这是后续所有问答的"上下文源头"。

#### 2. 多轮问诊式 AI 问答

交互流程：
- 用户输入问题。
- 系统判断"信息是否足够"——
  - 不足则**追问关键信息**（持续多久、有无伴随症状、最近是否换粮等）。
  - 足够则**结合档案 + 知识库召回**作答，附引用来源。
- 命中红线症状（呼吸困难、连续呕吐、尿闭、误食巧克力/百合等）直接弹**送医提示**，不让 AI 自由发挥。

每轮对话存表，便于回溯与后续优化。

#### 3. AI 归纳的动态手册

- 知识库按场景分类，前端按场景出页面。
- 每页内容由 LLM 基于该场景下的知识条目实时归纳渲染（结果缓存，避免重复消耗 token）。
- 页面底部展示原始知识条目列表，便于用户点开看细节。

### 不做（明确边界）

- 图鉴/成长相册（留接口位）
- 虚拟养成数值
- 用户社区/UGC
- 多猫管理
- 医院推荐、电商导流
- 真人兽医介入

### 安全与边界规则（产品红线）

- **明确写明"AI 不能替代兽医"**，所有医学相关回答末尾固定提示。
- **红线症状清单**：触发即跳出"立即送医"模板，AI 不再尝试诊断。
- **知识库由 LLM 整理审核**，前端需明示"内容由 AI 整理，仅供参考"。

## 五、核心交互设计：多轮问诊数据流

```
用户发问
    │
    ▼
后端组装 Prompt
    - System: 产品定位 + 红线规则
    - User: 当前问题 + 档案 JSON + 历史对话
    │
    ▼
调用 LLM（带 Tool Use），三选一：
    ├─ ask_followup        → 返回追问，前端继续对话
    ├─ answer_with_kb      → 召回知识库 top-5 → 二次调用生成答案 + 引用
    └─ emergency_redirect  → 返回固定送医模板
    │
    ▼
存入对话历史表
```

MVP 阶段不引入 LangGraph 等工作流引擎，靠 LLM 的 tool use 能力自决策即可。后续复杂了再升级。

## 六、技术架构

### 选型

| 层 | 选型 | 理由 |
|---|---|---|
| 前端 + 后端 | Next.js 14（App Router）+ TypeScript | H5 首选、单仓单部署、未来可用 Taro 复用到小程序 |
| UI | Tailwind + shadcn/ui | 不丑、组件齐全、单人开发友好 |
| ORM | Drizzle | Schema 即代码，迁移简单 |
| 数据库 | 开发 SQLite / 生产 Postgres（Supabase 或 Neon） | 免费档够 MVP 跑 |
| 向量库 | pgvector（Postgres 扩展） | 一个 DB 管所有事，运维成本最低 |
| LLM 主 | Claude Sonnet 4.6 | 多轮对话与 tool use 稳定 |
| LLM 备 | DeepSeek 或通义千问 | 国内访问/成本兜底 |
| 部署 | Vercel + Supabase | 免费档够用，零运维 |

### 数据模型（MVP）

- `users`：用户表（邮箱/手机登录）
- `cats`：猫档案（属于一个 user，MVP 仅 1:1）
- `vaccinations` / `dewormings` / `weights`：档案子表
- `conversations`：对话会话
- `messages`：每条消息（role、content、tool_calls、引用的知识 ID 列表）
- `knowledge_chunks`：知识库 chunk（content、embedding、scenario、age_range、source、reviewed_at）
- `manual_cache`：手册页归纳结果缓存

### 不做的事

- 不拆微服务
- 不上 K8s
- 不引入 Redis（MVP 缓存放 DB 或内存即可）
- 不做独立的 Python RAG 服务（全 TypeScript 单仓）

## 七、知识库策略

### 来源优先级

1. **权威源（首选）**：AAFP/ISFM 指南、《猫病学》《猫行为学》中文译本、皇家/希尔斯临床科普、本土靠谱兽医的公众号长文。
2. **UGC（次选，仅补场景叙事）**：小红书/知乎，必须二次清洗，不直接入库。
3. **GPT deep research**：用来做"主题聚合"，输出结构化大纲，再人工/LLM 审。

### 场景分类（冻结 8 个）

- 新手第一周
- 免疫接种
- 体内外驱虫
- 日常饮食
- 行为异常
- 健康自查与就医判断
- 绝育与发情
- 洗护

### 入库流程

1. 每个场景跑一轮 GPT deep research，输出大纲 + 关键知识点。
2. LLM 按统一 schema 整理为 markdown 条目，每条带 frontmatter（场景、适用月龄、紧急度、来源、审核日期）。
3. 构建脚本切 chunk + embedding 入 pgvector。
4. 前端手册页直接读该场景下条目，让 LLM 实时归纳渲染（带 7 天缓存）。

### MVP 体量

每场景 10–20 条，总量 100–150 条，一周内可完成。

## 八、开发路线图（4 周 MVP）

| 周次 | 目标 |
|---|---|
| 第 1 周 | 技术栈搭骨架；跑通"档案 CRUD + 单轮问答（不带知识库）" |
| 第 2 周 | 知识库内容整理 + 入库 + 召回；单轮问答升级为带 RAG |
| 第 3 周 | 上多轮问诊（tool use 三选一）+ 红线规则 |
| 第 4 周 | 手册页（动态归纳）+ UI 打磨 + 部署上线 |

四周末交付一个可以让朋友用上的 H5。

## 九、商业化思考（远期，非 MVP 范围）

- MVP 阶段先免费，目标做用户量。
- 数据基建提前埋点：用户系统、调用次数记录、对话数据留存（合规前提下）。
- 远期可能方向（待验证）：
  - 订阅制（高级问答额度、健康报告）
  - 宠物用品广告/导流（需小心，避免影响信任）
  - 与线下兽医/医院合作转诊

## 十、待讨论的开放问题

1. **产品名 / 域名**：目前仓库目录叫 `pet`，需要确定中英文产品名。
2. **知识库审核风险**：LLM 一遍过快但有错答风险，是否需要在上线前找 1–2 位养猫资深用户/兽医做一次抽审？
3. **冷启动用户**：四周后第一批用户从哪里来？小范围邀请测试 vs 直接小红书发种草贴？
4. **数据合规**：用户档案、对话内容如何存储？是否需要在 MVP 就做隐私协议？
5. **品牌调性**：偏专业医疗向（克制、严肃）还是偏陪伴养成向（温暖、可爱）？这个会影响 UI、文案、配色全部决策。
6. **失败止损线**：什么情况下我们认为这个产品验证失败、需要转向？比如三个月无自然增长、或核心问答 NPS 低于某线。
