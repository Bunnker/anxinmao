# AI 生成形象 · 实施说明

> **状态:** v0.1 草稿 · 决策记录 + 实施边界 + prompt 策略
> **决定日期:** 2026-05-24
> **背景:** 原 CLAUDE.md 红线「不做 AI 生成宠物形象」(office-hours 审核留下)
> 被精确化为「仅用于头像 / 伴侣角色,不进入医学示意图」。本文档记录这次松动的
> **边界**与**护栏**。

---

## 一、为什么松动这条红线

**情绪轴需要它。** 产品主轴是「怕 → 安心」,而「我家猫出现在科普卡片角落」能
把这条情绪推得更深 —— 用户感知到「这个 App 认识我家猫」,而不是「这是一份通用
说明书」。

**但风险真实存在,所以不全松。** 把 AI 生成图塞进医学场景的风险:
- 画一只精神不错的猫去配「严重脱水」文字,信号反向
- 「牙龈苍白 vs 粉红」「呼吸急促 vs 平稳」「瞳孔大小」这种诊断细节,AI 卡通图
  很难精准还原差异
- 用户对自家猫形象的信任度天然高 —— 万一图错,信号被放大

→ 结论:**形象边界严格限定在「身份 / 陪伴 / 气氛」,绝不进入「症状 / 诊断 / 信号」。**

---

## 二、允许 vs 禁止的位置(护栏)

### ✅ 允许出现 AI 生成形象的位置

- 首页 greeting 区(问候语旁的小头像)
- 报告卡片右上角或角落(作为「这是关于你家猫的报告」的身份标识)
- 完成卡片 / 引导卡片的装饰角色
- 知识页**整体氛围插图**(场景级,不指向具体症状细节)
- onboarding 完成页的庆祝场景

### ❌ 禁止出现 AI 生成形象的位置

- **症状描述图**(如「猫呼吸急促时的姿态」「脱水的皮肤回弹示意」)
- **病征对比图**(「牙龈正常 vs 苍白」「瞳孔大小对照」)
- **急诊红档** 任何位置 —— 红档信息层只能用文字 + 三色 + icon,不混入插图
- **症状卡分诊流程**(symptoms / flow 各步)—— 流程内插图会分散注意力
- 任何兽医资料中明确给出的「视觉自检方法」(咳嗽页的「皮肤回弹」三项自检等)
  —— 这类视觉对照必须用真实临床示意图(如有)或纯文字,不用 AI 卡通

### 灰区(默认禁止,改用前先讨论)

- 报告 yellow / green 档内的「护理建议示意」—— 文字够用,加图反而稀释信息密度
- behavior 问答的对话头像 —— 出现概率不低,会不会让用户误以为「AI = 我家猫在跟我说话」
  从而过度信任答复?暂禁,后续单独评估

## 三、技术路线

### Provider 选型:通义万相(DashScope wanx2.1)

- **已有 key** —— `.env.local` 里 `DASHSCOPE_API_KEY` 复用即可,不引入新 provider
- **中文 prompt 友好** —— 不用做翻译中间层
- **卡通插画风格成熟** —— DashScope 模型对「扁平卡通」「儿童绘本」类风格有专门优化
- **成本** —— turbo 档约 ¥0.16/张,6 张 < ¥1;plus 档约 ¥0.4/张

模型:**首选 gpt-image-2 via codex CLI**(实测风格一致性 + 细节遵循度比 wanx 强一档,
且 codex 内置 image_gen 不需要 OPENAI_API_KEY)。wanx2.1-t2i-turbo / 直 OpenAI API
作为兜底。详见 §四 「Generator 选型(本批次实测)」。

### 通用配图(本批次的内容)

- **一次性生成,作为静态资产入仓**(`public/knowledge/*.webp`)
- 不走运行时 API —— 不烧 token、不延迟、不依赖网络
- 生成脚本 `scripts/generate-knowledge-images.mjs` 保留在仓库里,prompt 也提交,
  便于复现 / 重生成 / 后续调优
- 输出格式:webp(体积小,移动端友好)
- 尺寸:1024×1024(知识卡上展示约 60-80px,但留高分屏 retina 余量)

### 用户家猫的头像(后续批次)

- **运行时按需生成**,服务端 API 调 DashScope,密钥不进浏览器
- 存储:Vercel Blob(第一次给项目引入云存储)
- 缓存:生成后 URL 存 localStorage,无需每次再生
- 触发点:onboarding 流程加一步「给 {name} 一个卡通形象」(文字描述 + 可选拍照)
- 展示位置:严格按本文档 §二 的允许清单

## 四、本批次的 6 张知识页配图 · prompt 策略

知识页主题是「看着吓人但不必慌」—— 图的情绪基调统一是 **放松 / 安心**。
**所有 6 张图主体都是「正常状态下的猫」**,通过 **关联物件 / 关联场景** 锚定话题
(体温计 / 猫砂盆 / 创可贴 / 月光 / 主人怀抱 / 加湿器),**绝不画症状细节**。

### Generator 选型(本批次实测)

最终落地走 **gpt-image-2 via codex CLI** —— 比 wanx2.1 风格一致性强一档,
6 张明显像「同一套配套插画」而不是「6 种不同风格的猫」;细节遵循度也更高
(创可贴 / 体温计 / 加湿器都画对了,无需二次强化 prompt)。

通道:`codex exec <prompt> -s workspace-write`
- 让 codex agent 用它**内置的 image_gen 工具**生成,不直接 fetch OpenAI API
- 优点:用户层面零配置,不需要 OPENAI_API_KEY,不受 sandbox 网络限制
- prompt 文件:`/tmp/codex-image-prompt.txt`(参考 `scripts/generate-knowledge-images-openai.mjs` 里的 IMAGES 表)

兜底通道:
- `scripts/generate-knowledge-images.mjs` —— DashScope wanx2.1-turbo,需要 `DASHSCOPE_API_KEY`
- `scripts/generate-knowledge-images-openai.mjs` —— 直接调 OpenAI Image API,需要 `OPENAI_API_KEY`

### 共用风格 token

```
Flat cartoon illustration style, warm beige background,
gentle and healing mood, minimalist composition,
children's picture book quality, soft warm lighting,
generous whitespace, no text, no watermark.
```

英文 prompt —— gpt-image-2 / DALL-E 系列对英文 prompt 更稳;wanx 用中文等价版本。

### 6 张图的 prompt(主体内容)

| id | 主题 | 关联物件锚点 | prompt 主体 |
|---|---|---|---|
| fever | 发烧本身 | 体温计 + 主人手摸额 | 猫趴在凉爽地板上,主人手轻搭额头,旁边放体温计 |
| bloodInStool | 粪便少量血 | 猫砂盆 | 灰白短毛猫从猫砂盆走出,姿态健康,尾巴竖立 |
| fightAbscess | 打架小脓肿 | 大粉色十字创可贴 | 橘虎斑猫坐着,脸颊贴特别醒目的粉色十字卡通创可贴 |
| femaleHeat | 母猫发情夜啼 | 月光 + 抬尾扭动 | 母猫仰躺扭动,抬高尾巴,窗外月光夜空 |
| singleSeizure | 抽搐后恢复 | 主人怀抱 | 小猫安心地被主人抱在怀里,主人手抚摸,猫完全平静 |
| cough | 猫咳嗽 | 加湿器喷雾 | 猫趴窗台,旁边卡通加湿器喷柔和水雾 |

完整英文 prompt 在 `scripts/generate-knowledge-images-openai.mjs` 的 `IMAGES` 表。

### 为什么都画放松状态的猫

知识页本身的论点是「这些情况看着吓人 → 但不必慌」。配图情绪必须支撑这个论点 ——
画一只焦虑 / 痛苦 / 病态的猫,等于跟标题对着干,把用户拉回「怕」的情绪里。

→ **配图传达「放松」,文字传达「升级条件」。** 升级条件由文字承担,严格、明确;
配图只负责让用户的情绪先落下来。

### 「图中讲解症状」—— 用 HTML 叠图,不依赖 AI 嵌字

接入时遇到的真实诉求:用户希望「图与症状相符 + 图中讲解症状」,让科普卡更有教育
意义。如果用 AI(gpt-image-2 / wanx)在图里直接渲染中文标注:

- 中文文字渲染目前还不稳,十次里六七次出错字 / 缺笔画
- 信号文字一旦烧死在像素里,改文案 / 本地化 / 加无障碍 alt 都得整张重生
- 医学护栏(§二)被破:AI 可能把信号画反(精神不错的猫配「严重脱水」)

→ **正确做法:AI 出底图(放松场景),HTML/SVG 在图上叠「不必慌信号」chips。**

每张卡从 `why` 里抽出 2-4 个「这几项都正常 = 通常可以白天再约门诊」的关键信号:

| id | chips |
|---|---|
| fever | 会喝水 / 会吃饭 / 牙龈粉红 / 呼吸平稳 |
| bloodInStool | 精神好 / 能吃能喝 / 血量少 / 不是黑便 |
| fightAbscess | 精神好 / 能吃能喝 / 脓肿不增大 / 无发热 |
| femaleHeat | 嚎叫 / 打滚扭动 / 抬尾 / 找异性 |
| singleSeizure | 持续 < 几分钟 / 单次发作 / 之后行为正常 |
| cough | 牙龈粉红 / 呼吸平稳 / 皮肤迅速回弹 |

样式:绿色 ✓ + 白底圆角 chip,叠在图底部,带黑色 gradient 蒙层(从 55% → 透明)
保证在任何图上都可读。chips 是真实 DOM 文字 —— 可编辑、可本地化、屏幕阅读器友好。

结果:
- 「图文并茂」+「症状相符讲解」效果都拿到
- 医学护栏(§二)严格守住 —— AI 没参与生成任何信号文字
- 后期改文案 / 翻译 / 调色 都不用重生图

实现见 `src/app/knowledge/page.tsx` 的 `ItemCard` 组件。

## 五、需要事后审核的事项

1. **风格一致性** —— 6 张图分别生成,模型在不同 prompt 间可能跑出不同笔触 / 配色。
   先看出图效果,如果一致性差,后续可考虑:
   - 用统一 seed + style image 引用
   - 改用「设计师手绘 / 一次性 Midjourney 套图人工挑选」路线
   - 或一图多次生成挑最一致的
2. **是否需要 alt text 描述细节** —— 当前 page 没设 alt,得加(无障碍要求)
3. **图加进来后是否过载** —— 知识页本来就长,6 张图叠上去文字密度被冲淡,可能反而
   让用户不容易找到「升级条件」。出图后先 screenshot 看整体效果再定保留 / 调整
4. **后续头像功能** 上线前,这份文档要补:Vercel Blob 接入、生成 API 设计、
   onboarding 改造、cost 监控
