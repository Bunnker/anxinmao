# 桌宠设计 · 项目文档

## 1. 一句话定位与本文目的

「安心猫」桌宠，是分诊器首页里那只随风险退场的橘猫——它把「打开 App 时的怕」接住，让用户在选症状之前先松一口气；它不养成、不收费、不喧宾夺主，只做情绪缓冲与轻量入口。

本文档面向团队成员、未来协作者，以及未来回来接手的自己，目的是让读者一篇读懂：我们的桌宠是个什么产品、为什么这么设计、有哪些交互、守哪些红线、现在到哪了、接下来做什么。所有状态机、坐标、文案、资产、红线均以当前代码（`src/app/page.tsx`、`src/components/PetSprite.tsx`）和成文红线（`CLAUDE.md`、`docs/product/AI生成形象-实施说明.md`）为准。凡代码内注释与实际数值冲突处，一律以代码实际值为准，并在相应章节标注。

## 2. 产品背景与桌宠的角色

### 情绪轴：怕 → 安心

主产品是「猫不对劲 → 选症状 → 追问几个问题 → 红/黄/绿风险分级 + 护理建议」。新手用户打开它的时刻，往往是焦虑的：猫吐了、不吃了、呼吸不对。整条产品的情绪设计目标就是把用户从「怕」搬到「安心」。

桌宠承担的是这条轴的**起点缓冲**：进入首页，先看到一只在自家小院里漫步、晒太阳、洗脸、钻纸箱的橘猫，而不是一张冷冰冰的症状网格。它传递的潜台词是「这个 App 认识我家猫、和我站在一起」，从而降低进入分诊的心理摩擦。

### 为何要桌宠，以及它与主功能的关系

桌宠是**门面与情绪载体，不是主功能**。它的角色主次明确：**情感锚 > 信息枢纽 > 轻量入口**。它绝不做养成、不做玩具、不做电商。理性的就医入口（看病 / 问答 / 小知识）始终以明牌形式存在（下沉到底部 sheet 与右上图标），桌宠的「搭话」只做感性邀请，绝不替代或盖过这些入口。

设计上有一条核心纪律：**可爱不能稀释医疗严肃性**。这通过两个机制保证——退场机制（风险升级时橘猫让位给文字+三色+icon）与 AI 形象边界（卡通形象绝不进医学示意图，详见第 8 节）。

### 「守门橘猫」方向与现状

2026-06-14 的多 agent 调研定下产品方向为「守门橘猫」（克制工具优先），两个核心机制：

- **退场机制**：在场度由 `decideTier`（纯规则判级，redFlag 即时判红、LLM 不参与）控制——绿档卖萌陪伴 / 黄档收敛 / 红档完全退场。把屏幕让给硬信息。
- **信任管线**：平时摸猫、喂水、梳毛顺手记一条 `observe` 记录，出事进 `/symptoms` 时注入上下文，把「平时陪伴」兑换成「出事时更信它的分诊」。

分阶段刻意小步（需求证据仅 n=1）：第〇批纯修隐患（画像直达分诊 + 红档退场做成可测回归 + 文案去愧疚）；第一批 observe 轻记录 chip + 注入分诊；第二批才碰连续感（坚持 Finch 范式：零惩罚 / 零 streak / 零愧疚）。

**现状**：用户 2026-06-14 决定**暂缓产品方向落地，先专注桌宠交互完善**（梳毛 / 动效 / bug）。要落地随时从第〇批起步。注意——「守门橘猫」spec 里把「多猫切换 UI」列为禁区，但该条已于 2026-06-16 红线反转为「要做」（见第 8 节），此处以最新红线为准。退场机制与信任管线尚未在首页代码中接线，属于规划态，本文档以现状区分。

## 3. 桌宠形态总览

### 方案 C：院子沉浸

首页采用「C 院子沉浸」方案：一个占满屏幕（除底部 sheet 外）的全屏院子 stage，橘猫在里面自主漫游、与家具/道具互动，底部一张固定的上拉风格 sheet 承载理性入口。这是 2026-06 由 Maple/claude.ai design 重设计后落地的形态，分支 `pet-home`。

整体层叠（`src/app/page.tsx`）：

```
<main h-dvh max-w-[430px]>
├── 院子 stage (flex-1, 全屏沉浸)
│   ├── 背景 yard-bg-v2.webp (object-cover 铺满)
│   ├── 奶白蒙层 rgba(247,246,243,0.12)
│   ├── 内容层 (width=345 设计基准, scale 缩放)  ← 家具 / 猫 / 盖帧 / 搭话泡
│   ├── 左上问候条 (z-30)
│   ├── 右上两图标 (z-30)  ← 小知识💡 + 使用说明?
│   └── 道具栏 care-float (z-20)  ← 梳子 / 水瓶 / 逗猫棒
├── 底部 sheet (z-10, 固定高度)  ← 回访 / 看病CTA / 问答 / 最近 / 反馈 / 免责
└── TabBar (z-50, layout 全局渲染, 首页显示)
```

### 2.5D 伪透视

院子是平面背景 + 伪透视的「2.5D」效果，而非真 3D。通过纵深带（`YARD_DEPTH=140`）+ 近大远小缩放 + 画家算法排层实现：猫越靠院子前沿越大、z 越高；家具按 bottom 排序前后遮挡。详见第 4 节末与第 5 节坐标系统。

### 橘猫雪碧图

橘猫的所有「单体动作」（idle、跑、招手、睡、玩、喝水等）来自一张雪碧图 `public/pet/spritesheet.webp`，由 PetSprite 组件以 CSS background-position 逐帧播放。涉及「猫+物件」的复合动作（钻箱、挠板、梳毛、逗猫棒、洗脸）则用 gpt-image-2 出的多帧 webp「盖帧」叠在家具上播放，期间隐藏雪碧图。雪碧图 + 盖帧两套机制共同构成完整的橘猫行为表现层（详见第 4、7 节）。

## 4. 行为系统：漫游状态机

桌宠的「活」感来自一台漫游状态机（`PetNudge` 内的 `roam` state + 调度 effect，`src/app/page.tsx` L826-1010）。猫在无人干预时自主切换状态：坐 → 掷骰子 → 散步去做点什么 → 回来坐着。`calm`（`prefers-reduced-motion` 或页面切到后台 `document.hidden`）或正在说话（`talk`）时暂停漫游。

### 4.1 雪碧图状态清单（PetSprite，17 行）

雪碧图 `spritesheet.webp` 实测 1664×3808 px，**8 列 × 17 行**，单格 192×208，格间 16px gutter（pitch 208×224，gutter 防缩放采样到邻格残影）。每行一个动作：

| 1 | running-right | [120×5] | loop | frames:[0,1,3,4,7]（跳过腾空帧）|
| 2 | running-left | [120×5] | loop | 同上跳帧 |
| 3 | waving | [260,260,260,280] | hold | holdFrame:2 招手定格 |
| 4 | jumping | [140×4,280] | hold | 跳一次落定 |
| 5 | failed | [180×7,240] | hold | 低落 8 帧 |
| 6 | waiting | [150×5,260] | loop | |
| 7 | working | [120×5,220] | loop | 专注（非跑步）|
| 8 | review | [150×5,280] | loop | |
| 9 | petted | [380,420,480,520,560,700] | hold | 本产品专属：被摸享受 |
| 10 | groom | [400,480,520,560,520,600] | hold | 本产品专属：洗脸 |
| 11 | nap | [550,620,680,620,660,720] | loop | 蜷睡呼吸 |

### 4.2 漫游状态清单（roam，20 个 kind）

roam 的 `kind` 联合共 20 个：`sit / stroll / groom / nap / wake / play / drink / hopin / hopout / box / greet / hop / scratch / drybowl / brushing / rug / pounce / sunbathe / washing`。初始 `{kind:"sit", x:4, y:58, facing:"right"}`。

| kind | 含义 | 时长 | 转移 |
|---|---|---|---|
| sit | 坐着待机 | 12–26s | 到点掷骰子选下一态 |
| stroll | 散步（≈35px/s 匀速 linear）| 距离/0.035 +80ms | 到达后接 `then`（drink 遇空碗→drybowl；box→hopin；否则 sit）|
| groom | 洗脸（雪碧图 groom 行）| 3600ms | → sit |
| wake | 起床（stretch/yawn/arch 随机）| 3300ms | → sit |
| greet | 招手 | 1600ms | → sit |
| hop | 原地开心蹦 | 1100ms | → sit |
| scratch | 挠抓板（盖帧）| 3200ms | → sit + scratch 猫语 |
| rug | 走到地毯 | 立即 | 选 washVariant，立即转 washing |
| washing | 洗脸盖帧 | 3600ms | → sit + WASH_TALK |
| pounce | 逗猫棒扑跳（盖帧）| 2400ms | → sit + play 猫语 |
| sunbathe | 阳光下趴眯眼（雪碧图 nap）| 6000ms | → sit + SUNBATHE_TALK |
| brushing | 梳毛盖帧 | 2600ms | → sit + BRUSH_TALK |
| play | 玩毛线球 | 4200ms | → sit + play 猫语 |
| drink | 喝水 | 4200ms | → sit，水量 -50 + drink 猫语 |
| drybowl | 凑空碗委屈（雪碧图 failed）| 2200ms | → sit + EMPTY_WATER_TALK |
| hopin | 跳进箱（帧 0→3）| 850ms | → box |
| box | 箱里东张西望 | 10–15s | → hopout |
| hopout | 爬出箱（帧 3→1）| 480ms | → sit + box 猫语 |
| nap | 打盹 | 18–32s | 睡醒随机 wake → wake |

> 代码内多处注释数值与实际定时器不符（washing 注释 240ms 实为 620ms 一拍；brushing 注释 ~3.4s 实为 2600ms；hopout 注释 ~800ms 实为 480ms），表中均以代码实际值为准。

### 4.3 sit 掷骰子调度

sit 到点后 `roll = Math.random()`，按累积阈值选下一态：

| 阈值 | 行为 | 约占比 |
|---|---|---|
| roll < 0.35 | stroll 散步（需 maxX>120 且距离≥40）| 35% |
| < 0.46 | groom 洗脸 | ~11% |
| < 0.54 | greet 招手 | ~8% |
| < 0.61 | hop 蹦跶 | ~7% |
| < 0.67 | sunbathe 晒太阳（走到 SUN_SPOT{190,30}）| ~6% |
| < 0.70 | nap 回窝睡 | ~3% |
| < 0.78 | play 玩球 | ~8% |
| < 0.86 | drink 喝水 | ~8% |
| < 0.93 | scratch 挠板 | ~7% |
| 否则 | box 钻箱 | ~7% |

需要走过去的态先经 `walkTo()`：距离 ≤24 直接做事，否则先 stroll 带 `then` 接续动作。调度用单个 `setTimeout`（返回值清理），每次 roam 变更触发 effect 重排。摸猫（`petTheCat`）、点家具（`goInteract`）会打断漫游。

> 代码内 sit 分支注释写「35/15/20/10/10/10」与实际阈值不一致，以阈值为准。

### 4.4 2.5D 深度算法

- `YARD_DEPTH=140`：纵深带，y（=bottom 偏移）0=最前沿、140=最里。
- `scaleOf(y) = 0.92 + ((140-y)/140)*0.14`：近大远小，范围 0.92（最里）~1.06（最前）。
- `zOf(bottom) = 100 - round(bottom)`：越靠前 z 越大。家具 zIndex = lifted?200 : rug?1 : zOf(bottom)。
- 猫 zIndex = `max(zOf(roam.y), 150)`——猫恒为前景，漫游到深处也不被家具盖住；地毯 rug 是「地面物」z 钉死 1，让猫站在毯上。
- 近大远小实现：猫容器 `translate(x, -y) scale(scaleOf(y))`，`transformOrigin: 50% 100%`（底中心缩放），家具同理按 bottom 排层。
- 活动边界：x∈[0, maxX=261]（345 基准宽 - 84 猫宽），y∈[0,140]。stroll 时 y 受斜率约束（横向必须主导，maxDy=max(8,|dx|/2)），防侧面猫纯纵移穿帮。

## 5. 交互设计

桌宠的用户主动交互有六类。所有院子坐标基于设计基准宽 `YARD_BASE_W=345`，外层按 `contentScale = min(1, yardW/345)` 等比缩放。

### 5.1 摸摸（点猫）

猫精灵外包一层 `<button aria-label="摸摸{猫名}">`。点击触发 `petTheCat()`：若正在散步，先就地停下变 sit；随机说一句 PET_TALK；随机演 petted（60%）或 groom（40%）动作；`touch.n` 递增传给 PetSprite 的 `playKey`，让连续摸每次都有反应（从第 0 帧重播）。3800ms 后清掉气泡和动作。

### 5.2 道具栏 care-float（梳子 / 水瓶 / 逗猫棒）

院子左下角三个毛玻璃道具按钮（52×52，`bottom: safe-area+32px`）。长按拖到目标命中触发：

| 道具 | 图标 | 拖到 | 触发 | 猫进入 |
|---|---|---|---|---|
| brush 梳子 | brush.webp | 猫身上 | brushTheCat() | brushing（随机 back/belly）|
| bottle 矿泉水瓶 | bottle.webp | 水碗 | setWater(100) + 猫跑来喝 | drink |
| wand 逗猫棒 | wand.webp | 猫身上 | pounceForWand() | pounce |

拖拽用 pointer 事件：按下立即 `setCarried` + `setPointerCapture`，移动更新跟手浮图（`position:fixed` rotate(-12deg)），松手做命中判定（cat 用 catRef rect，bowl 用 querySelector 找水碗按钮）。没命中则道具弹回工具栏。

### 5.3 家具拖拽

院子 6 件家具（`YARD_ITEMS`）全部可拖可点：

| key | alt | 初始 left/bottom/w | 点击→态 |
|---|---|---|---|
| bed | 猫窝 | 0/92/88 | nap |
| box | 纸箱 | 257/88/88 | box（→hopin→box→hopout）|
| bowl | 水碗 | 25/46/44 | drink |
| yarn | 毛线球 | 240/0/36 | play |
| scratch | 猫抓板 | 124/100/78 | scratch |
| rug | 小地毯 | 100/9/104 | rug（→washing）|

**点击**家具：猫 walkTo 锚点再做事（说话/被摸中不接单）。**拖拽**：长按 `LIFT_MS=400ms` 拿起（拿起前移动 >8px 当滑动放弃），屏幕位移除以缩放比换回基准坐标，left clamp 到 [0,345-w]、bottom clamp 到 [0,100]，拿起视觉 scale(1.08)+drop-shadow、z 升 200。布局持久化到 localStorage `yardLayout:v1`（全局一份，院子是共享的家，不分猫）。

**水碗水量**：`waterLevel` 0~100 存 `yardWater:v1`，碗图按水量换三态（>50 满 / >0 半 / =0 空）。喝一次降 50，空碗时猫凑前演 drybowl 委屈。

### 5.4 搭话泡

搭话泡（`pet-bubble`）仅在「摸猫」或「互动完成」时出现，实时跟随猫身体——算出猫的视觉左右边缘，比较两侧空间，贴空间更大一侧（泡宽 clamp [96,240]），尾角朝下指猫。**搭话泡不可点、无 href**，就是个展示用 `<div>`，只放一句文案。看病/问答入口不在泡上，已全部迁到底部 sheet。

> 一个工程注意点：帧态期间（box/hopin/scratch/brushing/pounce/washing 等）猫精灵**不卸载只隐藏**（opacity:0）。原因是卸载后切回 stroll 重新挂载的 div 没有上一帧 transform 作 CSS transition 起点，会从家具位置「瞬移」到新目标。常驻挂载保证平移平滑。

### 5.5 浮层入口（右上图标 + 左上问候条）

院子层（不随内容缩放，z-30）：

| 入口 | 位置 | 动作 |
|---|---|---|
| 浮动问候胶囊 | 左上 | 无跳转（pointer-events-none）。CatFace(relieved 36px) + 衬线「{greeting()},{猫名}」+ 月龄/性别/毛色 |
| 小知识 💡 | 右上圆按钮 | `<Link href="/knowledge">`（灯泡描边 SVG）|
| 使用说明 ? | 右上圆按钮 | onClick 开 Guide 弹层（非跳转，衬线「?」）|

注：问候头像是 `<CatFace>` 组件画的猫脸，不是用户上传/AI 生成的真实头像；首页未读取 cat.avatar 字段。设计稿原有的「竖排三 pill（看病/问答/小知识）」已整组删除，看病+问答下沉 sheet、小知识上移右上。

### 5.6 底部 sheet

sheet 容器 `-mt-[30px]` 负边距盖院子下沿、`rounded-t-[28px]` 圆角顶、向上阴影。**有视觉抓手（36×4px 灰条），但 sheet 不可拖动、不可展开收起**——抓手是纯装饰，无任何拖动/展开手势绑定，`flex-none` 固定高度。

sheet 内容自上而下：

| 入口 | href | 说明 |
|---|---|---|
| 回访闭环卡（条件出现）| — | 最近一条 12h~7d 内、未写 outcome 的 triage 记录触发，三选一「好多了/已就医/还没好」|
| 看病 CTA（主，陶土红渐变）| `/symptoms` | 「猫不对劲?选症状看病」+ 内嵌红黄绿三点 + 「30 秒红黄绿就医建议」|

## 8. 设计原则与红线

桌宠的一切设计都受主产品红线约束。以下红线「不可妥协」（来自 office-hours 审核），桌宠章节重申其中与形象/情绪相关的几条：

1. **医疗措辞红线**：一律「建议是否就医」，绝不出现「诊断」。每屏底部固定「AI 整理 · 不能替代兽医」（首页 sheet 底 Disclaimer 即此）。桌宠搭话**绝不出诊断口吻**（spec 原话「绝不你猫是不是病了」），不制造焦虑。

2. **红旗症状急停**：呼吸困难、抽搐、大量出血、尿闭、误食等中途即可急停直跳红档。对应 5 个红线急停 group（digest/breath/blood/urethra/ingest）。

3. **风险三色是独立信号层**：红/黄/绿不从暖色品牌盘里调，颜色+图标+文字三重传达。令牌 `--red:#d92d20 / --amber:#b97900 / --green:#238146`，比正文更饱和。在首页，三色信号层被院子 `overflow-hidden + isolate` 与暖色背景隔离，绝不被院子装饰污染。

4. **AI 生成形象不进医学示意图**（桌宠最核心的约束）：用户家猫的卡通形象**仅用于头像/伴侣角色**（greeting、报告卡片角落等），**绝不进入医学示意图**（分诊症状描述、报告病征示意、知识页关键信号示意）。
   - 允许：首页 greeting 头像 / 报告卡片角落 / 完成卡 / 知识页整体氛围插图（场景级）/ onboarding 庆祝场景。
   - 禁止：症状描述图、病征对比图、急诊红档任何位置、症状卡分诊流程、任何「视觉自检方法」图。
   - 理由：画精神好的猫配「严重脱水」=信号反向；牙龈苍白/瞳孔差异 AI 卡通难精准；用户对自家猫形象信任天然高，图错则信号被放大。一句话：形象边界严格限定「身份/陪伴/气氛」，绝不进「症状/诊断/信号」。

5. **不做（商业/账号/社区/养成）**：不做登录墙、电商/导流（含医院推荐，改用地图 deep link）、社区。桌宠不做养成——**禁区**：streak / 经验值 / 商城 / 晒猫社区。坚持 Finch 范式：零惩罚 / 零 streak / 零愧疚。

6. **多猫切换（红线已反转）**：原「不做多猫切换 UI」已于 2026-06-16 反转作废，现为「**要做**」——支持多猫切换/添加/删除，切换条在 `/pets`，数据 `Store.cats[] + activeCatId`，删猫连带删 records。注：「守门橘猫」旧 spec 把多猫 UI 列为禁区，已被此红线推翻，以最新红线为准。

7. **排版红线**：中文不用斜体；衬线（Noto Serif SC）只用于标题/情绪文字，功能文字用黑体。首页衬线共 3 处（问候语、使用说明「?」字符、看病 CTA 主标题），其余功能文字均黑体。

设计审美方向（非「不可妥协」级）：「iOS 克制精致 + 暖奶白底」，陶土红点缀 `--accent:#b05a50`，暖奶白底 `--paper:#f7f6f3 / --surface:#ffffff / --ink:#1a1a18`。这是视觉审美不是平台限定，移动端 H5 跨浏览器一致访问。

## 9. 技术实现概要

- **技术栈**：Next.js 16（App Router）+ React 19 + TypeScript，Tailwind CSS v4，localStorage 持久化。无数据库、无登录。
- **桌宠渲染**：雪碧图（`PetSprite.tsx`）用 CSS background-position 平移单格 + 嵌套 setTimeout 逐帧步进；复合动作用 gpt-image-2 出的盖帧 webp，藏 live 精灵后按锚点 ping-pong 播放。图集模块级预载一次，未就绪/失败回退静态 PNG。
- **状态机**：漫游 + 互动逻辑全在 `page.tsx` 的 `PetNudge` 组件（约 L308-1530），单 setTimeout 调度、roam 变更触发 effect 重排。
- **2.5D**：纯 CSS transform（translate + scale）实现伪透视，画家算法（zOf 按 bottom 排层）做前后遮挡。
- **响应式适配**：院子内容层固定设计基准（width=345、height=280），用 ResizeObserver 测窗口宽，`contentScale = min(1, yardW/345)` 等比缩放 + `translateX(-50%)` 居中。窄屏缩小不裁家具，宽屏封顶 1 不放大。家具拖拽换算用同一缩放比 sx。`floorLift = round(yardH*0.12)` 让内容落到背景地面带。
- **背景图**：yard-bg-v2.webp 在内容层外 object-cover 铺满，不随内容缩放；滤镜 saturate(0.92) brightness(1.02) + 一层奶白蒙层融合品牌色。
- **持久化**：家具布局 `yardLayout:v1`（全局一份）、水碗水量 `yardWater:v1`，均落 localStorage；首帧用 defaultLayout 避免水合不一致，客户端 useEffect 读 localStorage 覆盖。
- **省电与可访问性**：`prefers-reduced-motion` 静止、`document.hidden` 暂停步进；所有交互元素有中文 aria-label。
- **层叠**：TabBar（layout 全局，fixed z-50）> sheet（z-10）> 院子浮层（z-30/20）> 院子内容（scale 层）。院子内 isolate 封住高 z 精灵，猫不会盖到 sheet 上。

## 11. 风险与未决问题

1. **方案方向暂缓 vs 交互完善的张力**：「守门橘猫」产品方向（退场机制/信任管线）2026-06-14 起暂缓，当前只做桌宠交互打磨。退场机制是化解「可爱稀释医疗严肃性」的核心护栏，长期不接线，桌宠在红档场景的「该退场而未退场」是潜在红线风险，需在落地分诊/报告页时一并接上。

2. **代码注释与实际数值大面积漂移**：PetSprite 头部尺寸注释（11 行/1536×2288）、多处定时器注释（washing 240ms、brushing 3.4s、hopout 800ms、摸猫 4.2s）、sit 分支占比注释均与代码实际不符。本文档已全部以代码为准，但注释漂移本身是维护隐患，建议清一轮。

3. **历史残留文案/概念**：INTERACT_TALK.rug 三条在当前调度下走不到（rug 直转 washing 说 WASH_TALK）；「心事泡/思考泡」在代码中已无实现但注释仍提及。属死代码/死文案，需确认是删除还是接回。

4. **资产规范欠债**：源 PNG 滞留 public/、盖帧变体不全。前者影响包体与规范，后者影响洗脸/梳毛动作的丰富度。

5. **多猫与院子的关系未定**：家具布局/水碗水量是「全局一份，院子是共享的家，不分猫」，而 `/pets` 是多猫数据。院子里那只猫的形象/雪碧图是否随 activeCatId 切换、还是固定橘猫吉祥物，当前首页代码用的是固定雪碧图 + CatFace 组件画的猫脸（非真实头像），多猫场景下的桌宠身份归属尚未明确。

6. **部署门槛**：所有让线上变化的操作须用户显式批准；pet-home 分支当前未 push/未部署，demo 链路尚未打通。
