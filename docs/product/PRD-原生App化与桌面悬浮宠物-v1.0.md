- **一句话**:把「安心猫」从网页 + PWA/TWA 推进为**真正的原生 App**(自带前端、能离线、为上架与原生能力铺路),并以 **安卓「桌面悬浮宠物」** 作为差异化特色功能。

---

## 1. 背景

「安心猫」(www.whatsupkitty.cn)是面向新手猫主的移动端应用:猫不对劲时选症状 → App 追问 → 输出红/黄/绿风险分级 + 护理建议,另有行为/养育轻量问答。情绪轴:**怕 → 安心**。

- **现状**:Next.js 16 (App Router) + React 19 + TS + Tailwind v4;已部署上线;**PWA + TWA apk(包名 `cn.whatsupkitty.app`)已完成并发布**。
- **后端约束**:`/api/*` 服务端路由握着 LLM 密钥(DeepSeek/通义),做分诊、行为问答、头像生成、匿名云同步。**密钥不能进客户端,后端必须保持托管。**
- **数据**:localStorage + 匿名 deviceId 云同步;**无登录、无数据库**(产品红线)。
- **市场**:国内为主、兼顾国际;医疗相关;**个人开发者(无公司主体)**。

**为什么再往前走**:现有 TWA 仍**依赖手机的浏览器引擎 + 实时加载线上站**,体感偏"快捷方式";用户希望要**真原生**——自带前端能离线打开、能上架应用商店、有原生能力(推送等)。而且新提出的「桌面悬浮宠物」**只有原生壳能做**,PWA/TWA 做不到——这反过来是走原生的有力理由。

---

## 2. 目标 / 非目标

**目标**
1. **原生化(Capacitor)**:前端打包进原生壳、自带资源能离线打开;后端 API 保持托管。安卓 + iOS 双端出包。
2. **安卓桌面悬浮宠物**(本期重点特性):桌宠能"走到屏幕上",在桌面/其他 app 上漫游、互动。
3. **为后续铺路**:可靠推送提醒(疫苗/喂药)、上架应用商店。

**非目标(本期不做)**
- iOS 桌面悬浮宠物(系统不允许,见 §5.8);
- 立即上架(资质/合规是独立长流程,见 §7);
- 登录墙 / 数据库 / 电商导流;
- 改动任何分诊流程与医学内容(红线见 §10)。

---

## 3. 目标用户与场景

- **用户**:刚养猫、对猫的异常会焦虑的新手猫主。
- **悬浮宠物场景**:把陪伴感从 App 内延伸到桌面——一只猫在桌面漫游、卖萌、点一下快捷回到 App。强化"陪伴 → 安心"的情绪轴,也是留存与差异化的抓手。

## 4. 技术战略:Capacitor(依据 deep-research)

**选型结论**:在「个人开发者 + 已有 Next.js/React 前端 + 国内外双市场」三重约束下,**Capacitor 是复用度最高、改造量最小**的路线。

| 路线 | 现有代码复用 | 工作量量级 | 结论 |
|---|---|---|---|
| **Capacitor** | 把现有 web 前端打进原生 WebView,业务逻辑/API client/校验/TS 类型直接迁移、**UI 不重写** | **数周级** | ✅ **推荐** |
| React Native / Expo | UI 必须用 RN 原生组件重写,逻辑层约 60% 共享 | 8–12 周级 | ✖ 重写 |
| Flutter | 整体 Dart 重写 | 8–12 周级 | ✖ 重写 |
| 继续 TWA/PWA | 无新增 | — | ✖ 满足不了离线/原生能力 |

> 信源:nextjs.org 官方静态导出文档 / pkgpulse / nextnative / Capgo / LogRocket 交叉印证(详见 deep-research 报告)。

**架构**
- **前端**:Next.js `output: 'export'` 静态导出 → `out/`(纯 HTML/CSS/JS)→ Capacitor 的 `webDir`。界面随 App **本地加载、能离线打开**;客户端路由跳转类似 SPA。
- **后端**:`/api/triage`、`/api/behavior` 等是 POST + 读 request body + 流式 LLM,属"依赖 Request"类——**官方明确不能静态导出,必须留托管服务器**(恰与"密钥不进客户端"约束一致)。App 作为客户端联网调它。
- **离线边界**:界面/资源离线可用;分诊/问答/头像等 LLM 功能仍需联网调后端(合理且不可避免)。

**已知坑(落地需处理)**:无 `generateStaticParams()` 的动态路由、依赖 Request 的 Route Handlers、Cookies/Rewrites/Redirects/Headers/中间件/ISR/默认图片优化等在静态导出下不支持。本项目 `/triage`、`/report`、`/behavior?c=<id>` 等动态路由需补 `generateStaticParams()` 或改纯客户端路由;`localStorage/window` 访问需 `typeof window`/`useEffect` 守卫;图片需自定义 loader。**具体改造量需在真实代码上跑一次 `next build`(output:export)实测报错清单**(见 §12 待解问题)。

## 5. 核心特性:安卓桌面悬浮宠物(本期重点)

### 5.1 范围
**仅安卓做真·悬浮桌宠;iOS 保持现在的院子内桌宠不变。**(平台不对等是 iOS 系统限制决定的,见 §5.8。)

### 5.2 行为(用户选定:复用现有动作)
浮在桌面/其他 app 上 → 漫游 + 偶尔播放**现有逐帧动作**(洗脸/逗猫棒/walk 等,资产 `public/pet/items/*.webp`)+ 点击互动。**最大化复用已画好的帧。**

### 5.3 渲染架构(用户选定:WebView 复用动作)
**清晰分工:**
- **原生(Kotlin)= 容器与运动**:悬浮窗(`TYPE_APPLICATION_OVERLAY`,需 `SYSTEM_ALERT_WINDOW` 权限)+ **前台服务保活**(安卓规定,后台不被杀)+ **漫游**(用 `WindowManager` 移动"小窗口"位置)+ 触摸(点/拖/边缘吸附)。
- **WebView = 那只猫**:小窗口里嵌透明 WebView,**复用现有逐帧动画代码**渲染猫(idle/walk/洗脸/逗猫棒)。
- **关键**:**小窗口跟随移动**(窗口仅猫大小、跟着猫走)→ 避免全屏遮挡,窗口外触摸自然落到下面的 app。

### 5.4 组件拆分
| 单元 | 职责 |
|---|---|
| `FloatingPet` Capacitor 插件(Kotlin) | JS 可调:`requestPermission / show / hide / isShown / isSupported`;管悬浮窗 + 服务 |
| 前台服务 | 保活 + 常驻通知 + 漫游循环(移动窗口) |
| 悬浮 View(原生) | 承载透明 WebView;处理点/拖/边缘吸附 |
| `floating-pet` web(复用) | 极简页面,复用共享猫组件,只画猫 + 动作 |
| App 内设置项 | 「桌面悬浮宠物」开关 + 权限说明(见 §5.7) |

### 5.5 复用重构(顺带改善)
把院子里"渲染猫 + 逐帧动作"那部分**抽成一个共享组件**,让 **院子(app 内)与悬浮层共用同一套动画代码**。两只猫永远一致,这也是"复用现有动作"真正落地处。

### 5.6 交互流程
- 设置里开启 → 查权限 → 未授权跳系统授权页(带说明)→ `show()` → 起前台服务 → 猫浮现 → 沿屏边漫游(移动时 WebView 播 walk,停下播 idle/偶尔洗脸)。
- **点猫 = 把安心猫切回前台**(猫=快捷回 App 入口,顺带播卖萌反应);**拖动 = 挪位置**;**长按 = 收回**。
- 关闭设置开关或长按收回 → `hide()` → 停服务 → 移除悬浮窗。

### 5.7 设置项:「桌面悬浮宠物」(仅安卓)
位于「我的 / 设置」:
- **主开关:让桌宠出现在桌面**(on/off)。首次开启 → 说明 → 请求悬浮窗权限 → 授权后浮现;**拒绝则开关弹回 off + 提示**。
- **记住偏好**:开关持久化,下次打开 App 若仍是 on,**自动把猫放回桌面**(粘性)。
- **子项**(默认值,可改):「点猫时切回 App」默认**开**;「开机自动出现」默认**关**(开机自启悬浮窗要额外权限、更耗电、被 OEM 限制,想要再单开)。
- **(可选)** 院子里加个快捷开关,非必须。

### 5.8 iOS 处理
**iOS 系统沙盒禁止 app 画在其他 app/桌面上面,无对应权限、Apple 也不会让上架。** 故:插件在 iOS 上 `show/requestPermission` 为 **no-op(`isSupported=false`)**;设置项在 iOS **隐藏 / 置灰标「仅安卓」**;iOS 用户的院子内桌宠**完全不变**。(iOS 的"桌面小组件"是静态近似、不能漫游,本期不做。)

### 5.9 边界 / 错误处理
- 权限被拒 → 友好说明、开关弹回、猫留院子里。
- **Android 14+ 前台服务须声明 FGS 类型** + 通知,否则崩;**OEM(小米/华为等)可能激进杀后台**,常驻通知 + 正确 FGS 能缓解但不能根治(如实告知用户)。
- WebView 悬浮层:透明背景、无滚动条、**本地加载**(Capacitor 已把前端打成本地资源,悬浮页也是本地、离线可用)。
- 点/拖区分(touch slop);`show/hide` 幂等。

## 6. 原生能力路线图(后续阶段)

- **推送提醒(疫苗/喂药)** —— 国内现实:**FCM 在中国大陆不可用**,标准解法是接 **个推 / 极光**:在线走服务商自建长连接、离线走厂商通道(小米/华为/荣耀/OPPO/vivo/魅族);iOS 走 APNs。**可不破"无登录"红线**:用设备级 `ClientID(CID)`(SDK 自动生成、独立于账号),后端只需新增"推送服务 + 匿名 deviceId↔CID 映射表"(可复用现有匿名同步思路)。
  > 信源:个推/极光官方文档(详见 deep-research 报告)。
- **相机深度集成** —— Capacitor `@capacitor/camera` 官方插件即可满足,无需为相机单独走 RN。
- **桌面小组件**(Android App Widget / iOS WidgetKit)—— **成熟度待评估**(deep-research 未取得权威信源,见 §12),通常需写原生代码/第三方插件桥接,列为后续可选。

## 7. 上架与合规(现实,依据 deep-research)

| 商店 | 难度 | 关键要求 |
|---|---|---|
| **Google Play(国际)** | 🟢 最易 | $25 一次性账号、隐私政策、Data safety 表、内容分级、target API 达标、Lighthouse PWA ≥80。**已具备 HTTPS/manifest/assetlinks**。 |
| **国内安卓商店**(华为/应用宝/小米/OPPO/vivo) | 🔴 最难 | **企业主体(营业执照)** + **ICP 备案** + **软著** + **工信部 App 备案**(独立于域名 ICP,号码不同:`-1` vs `-1A`,个人可办 App 备案但商店多要企业);医疗/宠物健康类**可能要额外资质**。**个人开发者无主体是主要门槛。** |
| **Apple App Store** | 🟠 不确定 | 原生壳(Capacitor)+ 补原生推送/相机后,比纯 TWA **更易过 Guideline 4.2「最低功能」**,但**不保证**(仍有带推送的 Capacitor App 被拒先例,关键是 app-like 体验);$99/年。 |

> 信源:阿里云/华为云 ICP·App 备案 FAQ、Apple 审核指南、MobiLoud 等(详见 deep-research 报告)。

## 8. 分阶段路线

- **Phase 0 · Capacitor 地基**:`next.config` 开 `output:export` → 修动态路由(`generateStaticParams`/改客户端路由)、`window/localStorage` 守卫、图片 loader → 接 Capacitor → 安卓 + iOS 双端跑通(能装、能用、界面离线打开、分诊/问答联网正常)。**先跑 `next build` 实测改造量。**
- **Phase 1 · 安卓桌面悬浮宠物(本期重点)**:抽共享猫组件 → `FloatingPet` 原生插件(悬浮窗 + 前台服务 + 漫游 + 触摸)→ `floating-pet` 复用 WebView → 「我的/设置」开关 → 安卓真机验。
- **Phase 2 · 推送提醒**:接个推/极光(在线+厂商离线通道)→ 后端加推送服务 + CID 映射 → 疫苗/喂药定时提醒。
- **Phase 3 · 上架准备**:隐私政策/合规材料 → Google Play 先行(门槛最低)→ 国内商店看是否注册主体 → iOS 视原生能力补足后再评估。

> 工作量量级:Capacitor 改造**数周级**,远低于 RN/Flutter 重写的 8–12 周级。

## 9. 产品红线(不可妥协,沿用现有)

- **医疗措辞**一律"建议是否就医",**绝不出现"诊断"**;红旗症状中途急停直跳红档;**风险三色(红/黄/绿)是独立信号层**,不从品牌盘调色。
- **不做**:登录墙、数据库、电商/导流、社区。
- **AI 生成形象**仅用于头像/伴侣角色,**不进医学示意图**。
- **悬浮宠物**:纯陪伴,文案不焦虑、不诊断、不碰风险三色。

---

## 10. 风险

- **桌宠**:OEM 杀后台 / 耗电 / 悬浮窗权限授予率低;Android 对悬浮窗政策持续收紧。
- **上架**:个人开发者无主体 → 上不了国内主流商店;医疗/宠物健康类可能要额外资质。
- **iOS 悬浮**:系统层面做不了(已接受,iOS 仅院子内桌宠)。
- **静态导出**:Next 16 动态路由改造量未实测,可能有隐藏坑。

---

## 11. 成功标准

- **Capacitor**:安卓 apk / iOS ipa 能装能用;界面离线打开;分诊/问答联网功能正常;复用现有前端、未重写 UI。
- **悬浮桌宠(安卓)**:真机能从设置放出来、桌面漫游、点击切回 App、设置控制开关生效、切后台不被秒杀、记住偏好。
- **(后续阶段)**:推送到达率、商店过审 —— 各自阶段单列标准。

## 12. 待解问题(open questions,来自 deep-research)

1. **桌面小组件在 Capacitor 下的成熟度**:有无成熟第三方插件,还是必须各平台手写原生再桥接?(本期非目标,但用户提过)
